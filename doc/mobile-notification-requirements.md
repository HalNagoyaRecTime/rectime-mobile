# モバイル通知機能 要件定義書

## 1. 概要

本機能は、Firebase Cloud Messaging（FCM）とバックエンド通知 API を利用して、rectime モバイルアプリで通知を受信、表示、管理する機能である。

FCM は即時通知の受信手段として扱い、通知履歴、未読件数、既読状態の正はバックエンド API から取得する。

アプリ起動中は画面内通知や通知ベルへ反映し、バックグラウンド中または未起動時は OS の Push 通知として表示する。

## 2. 目的

- ユーザーが重要なイベントを素早く把握できる
- アプリ起動中は通知状態を即時反映できる
- アプリ未起動時、またはバックグラウンド中でも Push 通知を受け取れる
- 通知一覧、未読件数、既読状態をバックエンドの保存状態と同期できる
- FCM 配信失敗や端末差異があっても、アプリ表示時に正しい通知状態へ復元できる
- FCM や OS 通知の制約が UI やドメイン表現へ漏れないようにする

## 3. 前提

- モバイルアプリは Kotlin Multiplatform / Compose Multiplatform で実装する
- 通知対象は認証済みユーザーのみとする
- 通知配信には Firebase Cloud Messaging を利用する
- Android / iOS を Push 通知の対象とする
- iOS では FCM の内部で APNs を利用する
- Desktop（JVM）は Push 通知の対象外とし、必要な場合は画面内通知のみを扱う
- バックエンドは以下の API を提供する
  - `POST /notification-tokens`
  - `DELETE /notification-tokens`
  - `GET /notifications`
  - `GET /notifications/unread-count`
  - `PATCH /notifications/{id}/read`
  - `PATCH /notifications/read-all`

## 4. 対象ユーザー体験

### 4.1 通知許可

- ユーザーはログイン後、設定画面または通知画面から Push 通知を有効化できる
- 初回ログイン直後に OS の通知許可ダイアログを自動表示しない
- 通知許可を求める前に、画面上で通知を有効化する操作を明示する
- 通知が拒否済みの場合は、OS 設定で再許可が必要であることを画面上で示す
- 端末または OS の制約により Push 通知が利用できない場合は、画面内通知と通知履歴のみ利用可能にする

### 4.2 フォアグラウンド通知

- アプリ表示中に FCM メッセージを受信した場合、通知ベル、未読件数、通知一覧へ反映する
- 重要度に応じて画面内バナー、トースト、または通知一覧内の強調表示を行う
- 通知クリック時は `linkUrl` がある場合のみ対象画面へ遷移する
- 遷移前、または遷移後に対象通知を既読化する
- FCM 受信内容だけで完全な状態を確定せず、必要に応じて通知一覧または未読件数を再取得する

### 4.3 バックグラウンド通知

- アプリがバックグラウンド中の場合、OS の Push 通知として表示する
- Push 通知にはタイトル、本文、重要度、遷移先情報を反映する
- Push 通知タップ時はアプリを起動し、指定画面へ遷移する
- アプリ起動後に通知一覧と未読件数を再取得する

### 4.4 アプリ未起動時の通知

- アプリ未起動時に通知を受信した場合、OS の Push 通知として表示する
- Push 通知タップでアプリを起動する
- 起動後、通知データまたは `linkUrl` をもとに対象画面へ遷移する
- 起動後にバックエンド API から通知一覧と未読件数を再取得し、状態を同期する

### 4.5 通知一覧

- ヘッダーの通知ベルから通知一覧画面を開ける
- 通知一覧には新しい順で通知を表示する
- 各通知にはタイトル、メッセージ、作成日時、重要度、未読状態を表示する
- 通知が存在しない場合は空状態を表示する
- 一覧取得中、取得失敗、再試行可能状態を表示する
- 通知一覧から個別既読と一括既読を実行できる

### 4.6 通知設定

- 設定画面で Push 通知の有効、無効状態を確認できる
- 通知を有効化すると FCM トークンを取得し、バックエンドへ登録する
- 通知を無効化すると現在端末の FCM トークンをバックエンドへ削除または無効化要求する
- 将来拡張としてカテゴリ別 ON/OFF を追加できる余地を残す

## 5. 機能要件

### 5.1 Firebase 初期化

- Firebase Messaging の初期化はモバイルアプリのインフラ責務として分離する
- Firebase 設定値はプラットフォームごとの設定ファイル、またはビルド設定から読み取る
- Firebase 設定値が不足している場合、通知機能は無効状態として扱い、アプリ全体は起動可能にする
- Android / iOS 固有の Firebase 初期化処理を共通 UI へ直接漏らさない

### 5.2 通知許可管理

- ログイン後、ユーザー操作によって通知許可をリクエストできる
- 通知許可状態として、未確認、許可済み、拒否済み、非対応を区別できる
- iOS では APNs / FCM の利用に必要な通知権限を取得できる
- Android 13 以降では `POST_NOTIFICATIONS` 権限を取得できる
- 通知許可が得られない場合でも、通知履歴と未読件数は利用できる

### 5.3 FCM トークン管理

- ログイン後、通知許可が得られた場合に FCM トークンを取得する
- 取得した FCM トークンを `POST /notification-tokens` へ登録する
- 登録時に `platform` として `ios` または `android` を送信する
- 登録時に端末識別用の `device_id` を送信できる
- ログアウト時、通知無効化時、またはトークン破棄時に `DELETE /notification-tokens` を呼び出す
- 同一端末で登録済みのトークンを重複登録しない
- FCM トークンの再取得、更新、失効に備えて再登録できる
- トークンは UI やドメインモデルへ直接露出しない

### 5.4 通知状態管理

- 通知一覧、未読件数、受信中状態、エラー状態をアプリ内で管理する
- アプリ起動時、ログイン復帰時、アプリ復帰時に未読件数を取得する
- 通知一覧画面表示時に通知一覧を取得する
- FCM 受信時は未読件数を増やすか、バックエンドから未読件数を再取得する
- 既読化後のローカル状態更新とバックエンド状態の再同期を行う

### 5.5 API 連携

- バックエンド API のレスポンスはアプリ境界で正規化する
- API の `notification_id`、`link_url`、`created_at` などの外部形式を内部モデルの `id`、`linkUrl`、`createdAt` へ明示的にマッピングする
- API 失敗時は通知 UI のみ失敗状態にし、アプリ全体の利用を妨げない
- 個別既読 API は成功後にローカル状態を更新する
- 一括既読 API は成功後に未読件数を 0 にし、一覧内の未読表示を解除する

### 5.6 通知タップ遷移

- 通知データに遷移先が含まれる場合、通知タップで対象画面へ遷移する
- `linkUrl` または関連リソース情報をアプリ内ルートへ変換する
- 遷移先を解決できない場合は通知一覧またはホームへ遷移する
- 通知タップ後、対象通知を既読化する

## 6. 通知モデル

モバイルアプリ内部では以下の形に正規化する。

```kotlin
data class AppNotification(
    val id: String,
    val type: String,
    val title: String,
    val message: String,
    val linkUrl: String?,
    val resourceType: String?,
    val resourceId: String?,
    val severity: NotificationSeverity,
    val isRead: Boolean,
    val createdAt: String,
    val readAt: String?,
)

enum class NotificationSeverity {
    Info,
    Success,
    Warning,
    Error,
}
```

## 7. API 要件

### POST /notification-tokens

FCM トークンを登録する。

主な利用タイミング:

- ログイン後
- 通知許可取得後
- FCM トークン更新時

リクエスト例:

```json
{
  "token": "fcm-token",
  "platform": "ios",
  "device_id": "device-id"
}
```

### DELETE /notification-tokens

FCM トークンを削除または無効化する。

主な利用タイミング:

- ログアウト時
- 通知無効化時
- 端末の利用停止時

### GET /notifications

ログインユーザーの通知一覧を取得する。

### GET /notifications/unread-count

ログインユーザーの未読通知件数を取得する。

### PATCH /notifications/{id}/read

指定した通知を既読にする。

### PATCH /notifications/read-all

ログインユーザーの通知をすべて既読にする。

## 8. 画面要件

### 8.1 ヘッダー通知ベル

- 未読通知がある場合は通知ベルにバッジを表示する
- 未読件数を表示できる場合は件数を表示する
- 未読件数が 99 件を超える場合は `99+` と表示する
- 通知ベル押下で通知一覧画面へ遷移する
- アプリ復帰時に未読件数を再取得する

### 8.2 通知一覧画面

- 現在の `NotificationsScreen` を通知一覧画面として拡張する
- 通知は新しい順で表示する
- 各通知にタイトル、メッセージ、作成日時、重要度、未読状態を表示する
- 未読通知と既読通知を視覚的に区別する
- エラー通知、警告通知は視覚的に識別できるようにする
- 通知が存在しない場合は空状態を表示する
- 一覧取得中、取得失敗、再試行可能状態を表示する
- 通知項目クリックで詳細遷移、または既読化を行う
- 遷移先がない通知は既読化のみ行う
- 一括既読アクションを提供する

### 8.3 設定画面

- 現在の `SettingsScreen` に通知設定セクションを追加する
- Push 通知の状態を表示する
- 通知の有効化、無効化の操作を提供する
- OS 側で通知が拒否済みの場合、設定アプリで変更が必要であることを表示する
- Push 通知が非対応、未設定、有効、無効の状態を区別して表示する
- 将来的にカテゴリ別 ON/OFF を追加できる余地を残す

## 9. 非機能要件

- FCM に依存したデータ形式を UI コンポーネントへ渡さない
- Firebase Messaging、通知権限、トークン管理、API クライアント、UI 表示を別責務として分ける
- 通知機能が失敗してもログイン後の通常操作は継続できる
- 通知許可状態やトークン登録状態を必要以上に永続保存しない
- OS 通知権限、FCM 初期化、トークン取得、API 通信に関するエラーは開発者が追跡できる形でログ出力する
- ユーザーに表示するエラー文言は、復旧操作が分かる内容にする
- iOS / Android の通知権限差異を共通 UI から扱いやすい形に吸収する
- FCM だけに依存せず、アプリ起動時と復帰時はバックエンド DB から再取得して整合性を保つ

## 10. MVP 範囲

初期実装では以下を対象とする。

- Firebase Messaging の導入
- Android / iOS の通知権限取得
- FCM トークンの取得
- FCM トークンの登録、削除
- フォアグラウンド通知の受信
- バックグラウンド Push 通知の表示
- 通知ベルの未読表示
- 通知一覧画面での通知一覧表示
- 未読件数取得
- 個別既読化
- 一括既読化
- 通知クリック時の画面遷移
- アプリ起動時、通知一覧表示時、アプリ復帰時の再同期

## 11. MVP 対象外

- 通知カテゴリ別 ON/OFF
- メール通知連携
- 管理者による一斉通知作成画面
- 通知テンプレート管理
- 通知履歴の詳細ページング
- Slack / LINE / 外部チャットツール連携
- Desktop 向け Push 通知

## 12. 実装方針

- Presentation は通知ベル、通知一覧画面、設定画面の表示と操作に限定する
- Application は通知許可、トークン登録、通知取得、既読化のユースケースを扱う
- Domain は正規化済み通知モデルと通知状態の判断のみを扱う
- Infrastructure は Firebase Messaging、OS 通知権限、HTTP API の詳細を扱う
- API レスポンスと FCM ペイロードは境界で内部モデルへ明示的に変換する
- 既存の `NotificationsScreen` は通知一覧画面として拡張する
- 既存の `SettingsScreen` には通知設定セクションを追加できるようにする

## 13. 受け入れ基準

- 通知を有効化すると FCM トークンがバックエンドへ登録される
- 通知を無効化すると FCM トークンの削除または無効化 API が呼ばれる
- ログアウト時に FCM トークンの削除または無効化 API が呼ばれる
- フォアグラウンドで通知を受信すると通知ベルの未読状態が更新される
- バックグラウンドで通知を受信すると OS 通知が表示される
- アプリ未起動時に通知をタップするとアプリが起動する
- 通知タップ時に指定画面へ遷移できる
- 通知一覧を開ける
- 通知一覧で未読通知と既読通知を識別できる
- 個別既読と一括既読ができる
- アプリ再表示後もバックエンドの未読件数と整合する
- 通知機能が利用できない環境でもアプリの基本操作は継続できる
